# 開発プラン

## 1. 目的

- GitHub Pro の契約範囲内で、私の趣味を紹介・蓄積できる静的サイトを作る
- 機密情報を外部に出さずに、安全に開発・運用できる状態を維持する
- 最初はミニマム機能で公開し、改善と拡張を繰り返して育てる
- Copilot を活用しながら、設計・実装・テスト・PR・ナレッジ蓄積まで回せる workspace を育てる

## 2. 前提条件

- frontend は React latest を採用する
- 利用するライブラリはメジャーなものを優先する
- ログイン画面は作らない
- backend と db は、必要性が明確になった時だけ軽量なものを導入する
- テストは単体テストと E2E テストを基本とする
- AI agents instructions と Skills など、継続開発しやすい土台も整備対象に含める

## 3. 提案プラン

### Plan A: 最短公開プラン

最小構成で早く公開し、以後の改善サイクルを回すことを最優先にする。

対象範囲:

- トップページ
- サイトのコンセプト紹介
- 趣味一覧ページ
- 趣味詳細ページ
- レスポンシブ対応
- 基本的な単体テストと E2E テスト
- GitHub 上での build と deploy の自動化

メリット:

- 最短で形になる
- backend/db なしで始めやすい
- 失敗コストが低く、改善サイクルに入りやすい

デメリット:

- 検索、タグ、更新履歴などの拡張機能は初回に入らない

### Plan B: 発信強化プラン

Plan A に加え、趣味ごとの情報発信を継続しやすい構成を初回から入れる。

追加範囲:

- タグ・カテゴリ設計
- お知らせまたは更新履歴
- 記事一覧または活動ログ一覧
- 簡易フィルタ機能

メリット:

- 継続更新しやすい
- コンテンツが増えた時に整理しやすい

デメリット:

- 初回実装が少し重くなる
- 情報設計を先に決める必要がある

### Plan C: 拡張前提プラン

将来の拡張を見据えて、初回からデータ構造や運用基盤まで整える。

方針:

- 機能の初回スコープは小さく保つ
- 構造、運用、テスト、ナレッジ蓄積は最初から拡張前提で整える
- 改善・拡張は小さなイテレーションで継続する

追加範囲:

- コンテンツスキーマ定義
- データファイル分離
- 軽量 backend/db の導入判断
- ナレッジ蓄積用の記録フォーマット整備
- AI 用 instructions / skills / 作業テンプレート整備

メリット:

- 後から構造が崩れにくい
- workspace を成長させやすい

デメリット:

- 初回で考えることが多い
- MVP 公開までの時間が延びやすい

## 4. 推奨方針

本 workspace では、最初から Plan C を採用する。

理由:

- AI agents instructions / Skills / ナレッジ蓄積を最初から整える、という README の意図に合う
- workspace 自体を継続的に成長させるには、後付けより初期設計で基盤を入れた方がよい
- 最新技術を取り入れつつ、改善・拡張の反復を回しやすい
- 静的サイトでも、コンテンツ構造、テスト、運用を先に整える価値が高い

採用方針:

- 初回から Plan C の構造を入れる
- ただし、1回の実装スコープは小さく保つ
- 各サイクルで 1 つか 2 つの改善テーマに絞る
- 公開可能な状態を維持しながら段階的に拡張する

つまり、最小機能の MVP を作る点は維持しつつ、MVP を載せる土台は Plan C にする。

## 6. 参考にする運用原則

### 6.1 Goal / Task 管理を先に定義する

- Goal は GitHub Issue で管理する
- Task は Goal に紐づく sub-issue として管理する
- Goal と Task はラベルで明確に区別する
- ステータスは GitHub Projects の Todo / In Progress / Done で管理する
- Copilot が自然言語で扱えるよう、instructions に運用ルールを明記する

### 6.2 plan-first で設計してから実装する

- 改善案は、いきなり実装せずに最初に plan-first で整理する
- plan-first では、実装ステップ、影響範囲、リスク、必要テストを明文化する
- 大きい改善テーマは、Task を分割してから実装に進む
- ユーザー確認後に、実装と PR 作成のフェーズへ進む
- 実装依頼と追加要望は、対象 Task issue のコメント欄に assign 前の記録として残し、実行記録を一箇所に集約する
- 実際の開始は、対象 Task issue を Copilot に assign して行う

### 6.3 GitHub Actions 上の ambient agent を前提にする

- build / test / deploy だけでなく、提案、レビュー、軽微修正にも将来的に拡張できる形にする
- 最初から全部自動化せず、ラベル駆動と定期実行を段階的に導入する
- 人間は方針選択、レビュー、マージに集中する

### 6.4 Agent / Subagent の設計制約を守る

- Coordinator が専門 agent を呼ぶ構成にする
- Nested Subagents の深さは浅く保つ
- 呼び出せる agent は明示的に制限する
- Subagent に必要な情報は、親が明示的に渡す
- 内部用 helper agent と、ユーザーが直接使う agent の役割を分ける

### 6.5 セキュリティと権限を分離する

- AI 推論用のトークンと、GitHub 操作用のトークンは分離する
- GitHub 操作権限は最小権限を基本とする
- 機密情報は repo に置かず、必要時のみ secrets で扱う
- 別 repo 参照が必要な場合だけ、例外的に権限を拡張する

### 6.6 instructions / skills も自己改善対象にする

- 成果だけでなく、失敗理由や手戻り原因も残す
- Issue / PR の採否や実行結果から、instructions / skills / テンプレートを改善する
- workspace そのものを継続的に学習させる

## 7. 実行フェーズ

### Phase 0: 開発基盤の整備 — 完了 ✅

- React + TypeScript + Vite の初期構成を作る
- lint / format / test の基本設定を入れる
- GitHub Actions で build と test を回す
- コンテンツスキーマとデータ配置方針を決める
- Goal / Task のラベル設計と GitHub Projects の運用ルールを決める
- Copilot 向け instructions と作業ルールの置き場を作る
- ナレッジ蓄積用の記録先と記録フォーマットを決める

### Phase 1: Plan C ベースの最小公開版を作る — 完了 ✅

- トップページを作る
- 趣味一覧ページを作る
- 趣味詳細ページを作る
- サイト全体のレイアウトとナビゲーションを整える
- コンテンツをスキーマに従って管理する
- 単体テストと最低限の E2E テストを入れる
- 公開までの build / deploy フローを通す（`deploy.yml` で GitHub Pages に自動デプロイ）
- plan-first で実装計画を確認してから着手する流れを定着させる

### Phase 2: 改善サイクルを回せる状態にする

- 改善テーマの選び方を定義する
- Goal と Task の粒度を運用しながら調整する
- plan-first と実装フェーズの役割分担を定着させる
- PR ごとに成果、失敗、次の仮説を残す
- instructions / skills / 作業テンプレートを更新する
- サイクルごとの Done 条件を固定する

### Phase 3: コンテンツと体験を拡張する

- タグやカテゴリを導入する
- 更新履歴や活動ログを追加する
- 検索やフィルタを必要に応じて追加する
- デザインを磨いて世界観を明確にする

### Phase 4: workspace を継続的に成長させる

- 定期実行で改善候補を提案する仕組みを検討する
- 過去の失敗と成果から次の改善候補を定期抽出する
- 再利用できる UI / データ / テスト資産を増やす
- AI agents instructions / skills / 知見の質を継続改善する
- 拡張が必要なら backend/db 導入の要否を判断する

## 8. Plan C ベースの改善・拡張サイクル

1. Copilot が、Goal に紐づく改善候補や新規 Task を複数提案する
2. ユーザーが、今回進める Goal / Task を選ぶ
3. ユーザーが、実装対象の Task issue コメント欄に採用候補、追加要望、制約、完了条件を書く
4. ユーザーが、その Task issue を Copilot に assign して開始する
5. Copilot が同じ Task issue 上に plan-first で、影響範囲、実装ステップ、リスク、必要テストを整理する
6. 問題がなければ Copilot が実装、単体テスト、E2E テスト、PR 作成、Copilot review 依頼まで行う
7. ユーザーが PR を確認してマージする
8. 成果、失敗、改善案、運用上の学びをナレッジに残す
9. 次のサイクルで、instructions / skills / 運用ルールも含めて改善する

## 9. 1サイクルで選ぶ改善テーマの例

- 新しい趣味コンテンツを追加する
- 一覧や詳細の見やすさを改善する
- タグ、カテゴリ、検索を追加する
- データ構造を整理して拡張しやすくする
- テストを増やして回帰を防ぎやすくする
- instructions / skills を改善して作業品質を上げる
- GitHub Actions による提案やレビュー自動化を強化する
- backend/db 導入が必要かを検証する

## 10. Definition of Done

- そのイテレーションのスコープが画面で確認できる
- build が通る
- 単体テストが通る
- 必要な E2E テストが通る
- PR が作成され、レビュー可能な状態になっている
- Goal / Task と実装内容の対応が追跡できる
- 機密情報を含まない
- 今回の知見がナレッジとして残っている
